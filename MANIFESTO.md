# MANIFESTO — なぜ「気をつける」と書いても意味がないか

## 1. 前提

Claude Code を 2〜3 週間使うと、必ず次のような失敗を踏む。

- 本番 URL が 12 時間古いままなのに「完了しました」と報告される
- 別セッションが同じファイルを上書きして変更が消える
- CI が green なのに本番が壊れている
- 「気をつける」と CLAUDE.md に書いた翌週に同じ事故が再発する

最初は「自分の指示が悪かった」と思う。
2 回目は「CLAUDE.md が足りなかった」と思って規則を増やす。
3 回目に「ルールを増やしても効かない」と気づく。

ここから先は、**ルールを書くこと自体が問題**だという話。

---

## 2. なぜ「気をつける」が効かないか

LLM はインストラクションを「文字通り狭く」解釈する。
人間にとっての「気をつける」は「文脈に応じて適切に判断する」だが、LLM にとっての「気をつける」は「該当箇所に気をつけたフラグを立てて完了」になる。

例えば次のような事故が実際に起きた。

> ある本番サービスでスケジューラーが失敗し続け、ユーザーから見るデータが 12 時間古いまま放置された。原因はコード修正が「push されずに滞留」していたこと。あるセッションがコードを編集したのに `git push` し忘れたまま終わり、次のセッション起動まで本番に届かなかった。

このとき CLAUDE.md には「セッションが終わる前に push を確認する」と書いてあった。
だが Claude は「コードファイルを変更したセッション」を毎回正確に分類できない。書いたルールが「人間や LLM の判断」を要求している時点で、本当に push が必要な変更を見落とす経路が残ってしまう。

「気をつける」は **判断を人間に押し付けている** だけで、機械的に強制する仕組みがない。
人間も毎回判断できないからルールを書いたのに、そのルールが「人間の判断」を要求している。これは設計として閉じていない。

---

## 3. 物理ガードとは何か

「気をつける」を「機械的に違反不能にする」に変換するもの。

3 つの実話で説明する。すべて実際に踏んだ失敗を汎用化したもので、作り話ではない。

### 失敗例 1: 「本番確認してから完了」が守られない

ある本番サービスで Lambda が timeout し、データが 14 時間古いまま気付かれずに放置された。
CLAUDE.md には「CI が green なだけでは完了ではない。本番 URL で確認すること」と書いてあった。
だが Claude は「テストが通った = 完了」と判定して報告し、人間も「CI が green なら大丈夫だろう」と通してしまった。

**物理ガード**: `done.sh` が本番 URL を curl で叩き、HTTP 200 を確認した時刻を `Verified: <URL>:200:<timestamp>` の形式でコミットメッセージに必須追加する。`commit-msg` hook が `feat:` / `fix:` / `perf:` コミットで `Verified:` 行の欠落を検出 → 物理 reject。

結果: 「本番確認しないと commit が物理的に書けない」状態になった。Claude も人間も「気をつける」必要がなくなった。

### 失敗例 2: 「同じファイルを 2 セッションで触らない」が守られない

別の事例では、同じファイルを 2 つの worktree から並走編集してコンフリクトが頻発した。
CLAUDE.md には「同じファイルを 2 セッションで触らない」と書いてあった。
だが Claude は別 worktree で起動した別セッションの存在を認識できない。ローカルファイルシステムを見ないので、別タブで何が動いているか分からない。

**物理ガード**: `WORKING.md` に「いま誰が何のファイルを触っているか」を 1 行ずつ宣言する。`pre-commit` hook が宣言なしの編集を検出 → reject。さらに `guards/session/check-concurrent-sessions.sh` が `git worktree list` で 2 件以上アクティブな worktree を検出 → セッション起動を物理ブロック。

結果: 「並走する物理的可能性がゼロ」になった。気をつける必要がない。

### 失敗例 3: 「気をつける」「注意する」と書いた瞬間に守られなくなる

ルール文書のレビュー時に「気をつける」「できるだけ」「注意する」という曖昧な動詞が紛れ込む。
これが入った瞬間、ルール全体が「読まれる前提」になる。Claude も人間も「ここは気をつける箇所」と認識しても具体的な行動には繋がらない。

**物理ガード**: `pre-commit` hook が `.md` ファイルの diff に対して「気をつける」「注意する」「できるだけ」「なるべく」をパターンマッチ → 検出時にコミット reject。違反箇所と「物理ガード（CI / hook / SLI）に置き換えてください」のメッセージを表示。

結果: 「気をつける」と書こうとした瞬間に物理的に止まる。書けないものは規範にならない。

## 4. なぜルールを書くこと自体が問題か

ルールを書くとき、人は「これで再発を防げる」と感じる。
だが、「ルールを書いた」と「再発を防いだ」は別の事象だ。

ルールが効くには次の 3 つが揃わないといけない。

1. **読まれる** — Claude が起動時に必ず読む位置にある
2. **正しく解釈される** — LLM が「狭く解釈」しても破綻しない明確さがある
3. **違反が検出される** — 違反した瞬間に物理的に止まる仕組みがある

このうち、人間が書くルールは（1）と（2）には届くが、（3）には届かない。
（3）を実装するには CI workflow か git hook か検査スクリプトが必要で、これらは「ルールを書く」より工数が大きい。

結果として、ルールは積み重なるが事故は減らない。
CLAUDE.md が 500 行を超え、誰も全部読まないまま、新しいルールが追加され続ける。

このリポジトリは、その悪循環を断ち切るために作られている。

---

## 5. 「物理化されていない対策」を見破る 4 つの問い

ルール変更を提案された時、次の 4 つの問いを当てる。1 つでも No なら、それは「気をつける」と等価。

| 問い | No なら |
|---|---|
| 違反した時にどの行で物理的に止まるか言えるか？ | 「気をつける」と同じ。書いた人の自己満足 |
| その物理ゲートは Claude が CLAUDE.md を読まなくても動くか？ | CLAUDE.md を読み忘れた瞬間に効果ゼロ |
| その物理ゲートを bypass する手順は明示されているか？ | 緊急時に強制 push で迂回できないと運用が回らない |
| その物理ゲートの効果は数値で測れるか？ | 効果が見えないものは「念のため」で積み上がる |

ルールを書く前にこの 4 つを通す。通らなかったら、書かない。
書かないという判断は怠惰ではない。書いたものが効くと錯覚するほうが事故を増やす。

---

## 6. このリポジトリで何ができないか

正直に書く。

- **完全な事故防止** — 物理ガードは 8 割の事故を防ぐが、人間判断が必要な領域（仕様の良し悪し・倫理的判断）は防げない
- **組織展開** — 1 人運用を前提に設計している。組織向けの監査・権限管理は持たない
- **他エージェント対応** — Claude Code 固有の hooks 設計に依存している。Cursor / Aider 等への横展開は別設計が必要
- **既存事故のロールバック** — 既に踏んだ事故は元に戻らない。次の事故を物理的に防ぐだけ

「効くもの」と「効かないもの」を分けて、効くものだけを実装する。
効くものは少ない。だから少ないままで提供する。

---

## 7. 設計原則の最小セット

このリポジトリ全体に通底する原則は 5 つだけ。

1. **テキストで縛らない、構造で縛る** — ルールは hook / CI / cron に landing するまで「未実装」と扱う
2. **1 機能 1 スクリプト** — 1 つの guards は 1 つの違反パターンだけ検出する。束ねた瞬間に改修コストが跳ねる
3. **50 行以内・5 行ヘッダー必須** — guards スクリプトはコメント込み 50 行以内。目的 / 根拠 / 期待効果 / トリガー / テストの 5 行ヘッダーを書けないなら採用しない
4. **失敗時 exit 1** — 終了コードで合否判定。標準出力で警告するだけのスクリプトは guards と呼ばない
5. **bypass フラグを 1 つ用意する** — 物理ガードには必ず緊急脱出口を作る。脱出口がないと運用が回らず、結果的にガード自体を撤去される

これだけ守れば、後は何を物理化するかの選択。
このリポジトリはその選択を 8 件分（並走 / 「気をつける」/ 削除済参照 / main 直 push 等）実装して同梱している。

---

## 8. 最後に

このリポジトリは、誰かに「正しい使い方」を強制するためのものではない。

「Claude Code を信頼できる状態に物理的に保ちたい」という同じ欲求を持つ人に、
試行錯誤を 1 個分省略してもらうための、ただの実装一式。

フォークして、削って、追加して、自分のプロジェクトに馴染ませてほしい。
うまく行かなかった部分があれば Issue で教えてもらえると助かる（実装が間違っているか、こちらの前提が違うか、どちらかが分かる）。

そして、「気をつける」と書きたくなった瞬間に、もう一度この MANIFESTO を読み直してほしい。
書く前に止まるのが、いちばん効く物理ガードだ。
