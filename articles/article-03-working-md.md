## こんな事故を踏んだことはないか

Claude Code を使い始めて 1〜2 ヶ月が経つと、「もっと効率を上げられるのでは」と思い始める。大きなリファクタリングを走らせながら、急に割り込んできた hotfix を別セッションで並走させる。それぞれが同じファイルを別の方向に直し、30 分後に PR を出すと、どちらかの変更が消えている。

1 回目は「自分の段取りが悪かった」と思って手動でマージした。2 回目は「今度は気をつけよう」と心がけたのに、また同じ事故が起きた。

汎用化した実例を 3 つ紹介する。

1. セッション A が `src/utils.ts` の関数を修正中。セッション B が同じ関数を別の方法で修正して先に push。セッション A が push すると、どちらのコードも壊れた状態でマージされた。
2. セッション A があるファイルを削除。セッション B は同じファイルをまだ参照していて、削除後にゾンビ参照が残った。CI が通って本番で落ちた。
3. セッション A と B が同じ YAML 設定ファイルに別のキーを追加。マージ時にインデントが崩れ、パースエラーで CI 全 fail。

3 つに共通しているのは「セッション A は、セッション B が何をしているか知らない」という点だ。

---

## なぜ Claude は他のセッションを認識できないのか

これは Claude のバグでも、使い方の問題でもない。LLM の設計上の制約だ。

Claude は「今見えているコンテキスト」で判断する。別タブで起動した別セッションが同じファイルを編集していても、そのセッションはコンテキストに入っていない。Claude から見ると「そのファイルの最新版はファイルシステム上の状態」であり、「別セッションが 10 分前から同じファイルを編集中」という情報はどこにも存在しない。

これは `CLAUDE.md` に「並走する際はお互いを確認すること」と書いても解決しない。Claude はルールを読んでいても、「別セッションを確認する」という手段を持っていない。確認すべき対象が、Claude のコンテキスト外にあるからだ。

前の記事（[「気をつける」と書いても意味がない理由](article-02-soft-language-guard)）で書いたように、「テキストで縛る」設計はここでも壊れる。必要なのは物理ガードだ。

---

## 解決策 1: `WORKING.md` で並走を宣言させる

発想はシンプルで、「今、誰がどのファイルを触っているか」をファイルに書かせる。OSS リポに含まれている `WORKING.md` がそれだ。

```markdown
## 現在着手中

| タスク名 | 種別 | 変更予定ファイル | 開始 JST | needs-push |
|---|---|---|---|---|
| utils.ts 境界値修正 | [Code] task-utils-fix | src/utils.ts, tests/utils.test.ts | 2026-01-15 14:30 JST | yes |
```

ルールは単純だ。

- セッション開始前に `cat WORKING.md` を読む。自分が触ろうとしているファイルが他の行にあったら待つ。
- 行を追記してから着手する（「宣言なし編集禁止」が物理ルール）。
- 完了したら行を削除して push する。
- 開始から 8 時間が経過したエントリーは `session_bootstrap.sh` が自動削除する（クラッシュ後の stale エントリーを手動で掃除しなくて済む）。

ただし、これだけでは書き忘れる。人間が「気をつける」形の運用は、セッションが増えるほど崩れる。次のステップで物理化する。

---

## 解決策 2: `check-concurrent-sessions.sh` でセッション起動を物理ブロック

`session_bootstrap.sh`（セッション開始時に必ず実行するスクリプト）の中から、次のスクリプトを呼び出す。

```bash
#!/usr/bin/env bash
# 目的: 同時コードセッション 2 件以上を物理ブロック
# 根拠: 複数 worktree が同じファイルを編集してコンフリクト・上書き事故が頻発
# 期待効果: 並走違反を 0 件にする
# トリガー: session_bootstrap.sh 呼び出し時（セッション開始時）

set -u

# bypass: 緊急時のみ ALLOW_CONCURRENT_CODE=1 で escape 可能
[ "${ALLOW_CONCURRENT_CODE:-0}" = "1" ] && exit 0

REPO="${REPO:-$(git rev-parse --show-toplevel 2>/dev/null || echo "")}"
[ -d "$REPO" ] || exit 0

WT_DIR="$REPO/.claude/worktrees"
[ -d "$WT_DIR" ] || exit 0

NOW=$(date +%s)
WINDOW=$((30 * 60))  # 30 分以内に更新があった worktree を active とみなす

active=0
while IFS= read -r wpath; do
  [ -d "$wpath" ] || continue
  m=$(stat -c %Y "$wpath" 2>/dev/null || stat -f %m "$wpath" 2>/dev/null || echo 0)
  if [ $((NOW - m)) -lt "$WINDOW" ]; then
    active=$((active + 1))
  fi
done < <(git worktree list --porcelain | awk '/^worktree /{print $2}')

if [ "$active" -ge 2 ]; then
  echo "❌ コードセッション並走違反: active worktree ${active}件 (≥2)"
  echo "  対応: 他セッション完了待ち / bypass: ALLOW_CONCURRENT_CODE=1"
  exit 1
fi
exit 0
```

動作のポイントを 4 つ挙げる。

1. **`git worktree list` で全 worktree を列挙** — 現在のリポジトリに紐づく worktree 一覧を取得し、30 分以内に更新があったものを「active」とみなす。
2. **2 件以上で `exit 1`** — `session_bootstrap.sh` が起動失敗するので、セッション自体が物理的に開始できない。
3. **`ALLOW_CONCURRENT_CODE=1` の bypass を用意** — 脱出口がないと、急な hotfix が必要なときに詰まる。緊急時の迂回路を最初から提供しておくことで、ガード全体が「撤去される」リスクを防ぐ。
4. **エラーメッセージに bypass 方法を書く** — 「なぜ止まったか」と「どうすれば通るか」を止まった瞬間に表示する。記憶に頼らない設計。

## なぜ「並走禁止」ではなく「並走宣言制」にするか

最初は「コードセッションは 1 件だけ」という完全禁止ルールを考えた。しかし実運用で無理が出た。

リファクタリングの PR レビュー中に本番でバグが出て hotfix が必要になる。別の PR を待ちながら、依存しない別ファイルの作業を進めたい。こういったケースで完全禁止にすると、開発そのものが止まる。

「宣言制」の設計は、**並走自体は許可するが、同名ファイルの編集は宣言によって排他制御する**。`WORKING.md` に行があるファイルは触れない、ということが双方に見える状態を作る。物理的に「衝突が起きうる組み合わせ」だけを禁止して、それ以外は自由に動ける。

`ALLOW_CONCURRENT_CODE=1` の bypass も同じ発想だ。ガードは「必ず守らなければならない壁」ではなく「自分がどこにいるかを教えてくれる標識」として使う。bypass した事実が git のコミット履歴やログに残るので、後から振り返れる。

---

## 効果と限界（正直に言う）

このガードを導入してから、並走に起因するコンフリクト PR は月に複数件あったものがゼロになった。`ALLOW_CONCURRENT_CODE=1` の bypass 使用は月に 1〜2 件ほどで安定している。

ただし完璧ではない。

- `git worktree` を使っていない運用（同じディレクトリを別タブで 2 つ開くなど）では検出できない。別途プロセス検出が必要になる。
- Claude が `WORKING.md` への行追記を忘れた場合、このスクリプトは止められない。宣言なしの編集を `pre-commit` hook でも reject する設計が追加の防衛線になる（BOOTHフルキットに含まれている）。
- `WORKING.md` の更新自体を誰も push し忘れると、他セッションから「空き」に見えてしまう。「宣言したら即 push」というルールを bootstrap スクリプトのチェックと合わせて運用する。

記事①で書いた通り、物理ガードは事故の 3〜4 割を防ぐ防衛線で、万能ではない。それでも「0 件から何件防げるか」を測りながら運用を育てていくのが正しいアプローチだと思っている。

## 連載まとめ — 物理ガードという考え方

この連載で 3 つの物理ガードを扱った。

- **記事①**: 「Claude が instructions を無視する感覚は正しい」— 物理ガードという考え方の導入と、全体設計の概観
- **記事②**: 「`気をつける`と書いても意味がない理由」— `check-soft-language` フックで、ソフト言語を物理的にコミットできなくする
- **記事③（本記事）**: 「同じファイルを 2 つのセッションが踏み荒らす事故をゼロにした」— `WORKING.md` + `check-concurrent-sessions.sh` で並走を宣言制に

3 つを合わせると、Claude Code 運用で踏みやすい事故の多くに防衛線を張れる。ただしこれは「最低限の防御線」であって「完成した安全設計」ではない。自分のプロジェクトで実際に踏んだ事故に合わせてガードをカスタマイズしていくのが、長期的に効く運用だと思っている。

さらに深く対策したい人には、30 件の実失敗記録・CI workflow 一式・git hook フルセットを詰め込んだフルキットを用意している。
