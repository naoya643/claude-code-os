## ルール文書に「気をつける」が紛れ込む瞬間

`CLAUDE.md` や `docs/rules/` をレビューしていて、こういう一文に遭遇したことはありませんか。

> ファイル削除時は依存箇所も合わせて修正することに**注意する**。

> 同名のセッションが並走しないよう**気をつける**。

> 本番反映後は**できるだけ**早く動作確認を行う。

書いた本人は「再発防止策を 1 つ積んだ」つもりです。レビューも通ります。CLAUDE.md の行数は増えていきます。

ですが、こうした行は書いた瞬間に守られなくなります。前回の記事（[Claude が instructions を無視する感覚は正しい](claude-ignores-instructions)）で「物理ガード」という考え方を紹介しましたが、今回はその最初の 1 つ ─ **「気をつける」「注意する」「できるだけ」を物理的に書けなくする pre-commit hook** ─ を 5 分で導入できる形で書きます。

---

## なぜ「気をつける」がルールを無効化するか

「気をつける」は **動作を伴わない動詞** です。書いた人も読んだ人も、「具体的に何をいつどうやってやるか」を決めずに会話を終わらせられます。

人間が読むと「重要な箇所だな」と認識します。ですが「気をつける」の中身を毎回手動で展開できる人は実在しません。次の週にはもう忘れています。

LLM が読むと、もっとはっきり壊れます。Claude は「気をつける」を **該当箇所に『気をつけた』フラグを立てて完了**、として処理します。「依存箇所も合わせて修正することに注意する」と書いてあれば、依存箇所を grep せずに「注意しました」と返してきます。判断の精度を上げる呪文ではなく、**ただのチェックボックス**として消費されます。

結果、ルールは「書いた人の自己満足」になります。発火回数 0・bypass 回数 0・効果計測不能。書いてないのと同じ状態です。

代表的な「無効化動詞」を整理するとこうなります。

| 動詞 | 何を要求しているか不明 | 物理化案 |
|---|---|---|
| 気をつける / 気を付ける | 何を / どう / いつ | 該当箇所を CI で grep |
| 注意する | 同上 | 同上 |
| できるだけ | どの程度（90%？50%？） | 閾値を数値で書く |
| なるべく | 同上 | 同上 |
| 努める | 達成 / 未達成の判定不可 | 数値目標 + SLI |
| 心がける | 行動が定義されていない | 該当行動を script 化 |

共通点は「**主語が誰で、いつ、どこまでやれば完了か**」が決まらないことです。これは物理ガードと真逆の発想で、書いた瞬間に運用が破綻します。

## 物理的に書けなくする実装（コピペで動く）

`pre-commit` hook で `.md` の追加行をスキャンし、無効化動詞が含まれていたらコミットを reject します。次のスクリプトを `guards/commit/check-soft-language.sh` に保存してください。

```bash
#!/usr/bin/env bash
# 目的: 「気をつける」「注意する」等のソフト言語を検出してコミットをブロック
# 根拠: ソフト言語は書いた瞬間に対策完了と錯覚し、物理ガードが配備されない
# 期待効果: ソフト言語入り .md commit を 0 件にする
# トリガー: pre-commit hook
# テスト: tests/guards/test-check-soft-language.sh

set -u
[ "${ALLOW_SOFT_LANGUAGE:-0}" = "1" ] && exit 0

TARGETS=$(git diff --cached --name-only --diff-filter=AM | grep -E '\.(md|markdown)$' || true)
[ -z "$TARGETS" ] && exit 0

PATTERN='気をつけ|気を付け|注意する|注意します|できるだけ|なるべく|努める|努めます|心がける'

fail=0
while IFS= read -r f; do
  hits=$(git diff --cached -- "$f" | grep -E "^\+[^+]" | grep -E "$PATTERN" || true)
  if [ -n "$hits" ]; then
    echo "❌ '$f' にソフト言語が含まれています:"
    echo "$hits" | sed 's/^/    /'
    fail=1
  fi
done <<< "$TARGETS"

if [ "$fail" = 1 ]; then
  echo ""
  echo "物理ガードに置き換えるか、緊急 bypass:"
  echo "  ALLOW_SOFT_LANGUAGE=1 git commit ..."
  exit 1
fi
exit 0
```

このスクリプトで「効くようにする」ためのポイントが 5 つあります。

1. **`set -u`** — 未定義変数を即エラーにします。`PATTERN` を typo した時に「無検出のまま PASS」する事故を物理的に防ぎます
2. **`ALLOW_SOFT_LANGUAGE=1` の bypass** — 脱出口がないと運用が回りません。引用や説明文で意図的に「気をつける」を書きたい場面は必ず出ます。**脱出口を最初から提供しないと、ガードごと無効化（hook 削除）されます**
3. **`--diff-filter=AM`** — 追加 / 変更されたファイルのみ対象。既存ドキュメントは検査しません（ガード導入時に大量 reject が出ないように）
4. **`grep -E "^\+[^+]"`** — diff の追加行だけ（`+++` ヘッダーを除外）。既存行のソフト言語は触らず、**新規に持ち込もうとした分だけ止める**設計
5. **失敗時 `exit 1`** — pre-commit hook はこの exit code で commit 自体を reject します。「警告だけ出す」設計は無意味（読み飛ばされるだけ）

## pre-commit hook へのインストール

3 コマンドで終わります。

```bash
# 1. スクリプトを配置
mkdir -p guards/commit
# (上記スクリプトを guards/commit/check-soft-language.sh として保存)
chmod +x guards/commit/check-soft-language.sh

# 2. pre-commit hook を作成
cat > .git/hooks/pre-commit <<'EOF'
#!/bin/bash
set -u
bash "$(git rev-parse --show-toplevel)/guards/commit/check-soft-language.sh" || exit 1
EOF
chmod +x .git/hooks/pre-commit

# 3. 動作確認
echo "今後は気をつけて書きます。" >> README.md
git add README.md
git commit -m "test: try to commit soft language"
# → ❌ '<file>' にソフト言語が含まれています: で reject されれば成功
```

最初の reject 体験まで 5 分です。reject されたら `README.md` の差分を捨てて、書きたかった内容を **物理ガードに置き換える** のが本来の流れです。「今後は気をつけて書きます」と書きたくなった瞬間、それは `pre-commit` か CI で検出すべきパターンが見つかった瞬間でもあります。

## ガードを増やしすぎないための採用基準

「効いた」と感じると、似たガードを次々追加したくなります。ですが物理ガードを増やしすぎると、**自分のリポジトリで開発できない状態**になります。コミット 1 回で 10 種類の hook が走り、bypass フラグだらけのコマンドラインになり、最終的に全部 `--no-verify` で迂回されるのが見えています。

新しいガードを追加する前に、次の 4 項目を全部 ✅ にしてから採用してください。1 つでも No なら、まだ採用するタイミングではありません。

1. **違反した時にどの行で物理的に止まるか言えるか？** ─ 「pre-commit hook で reject される」「CI workflow が exit 1 で落ちる」など、止まる場所を 1 行で説明できること
2. **その物理ゲートは Claude が CLAUDE.md を読まなくても動くか？** ─ ルール文書への記載が前提のガードは「物理」ではなくただのソフト規律です
3. **bypass する手順は明示されているか？** ─ `ALLOW_*=1` のような環境変数ベースの脱出口を、エラーメッセージ自体に書いておくこと
4. **効果は数値で測れるか？** ─ 発火回数・bypass 回数・false positive 回数を週次で観測できること

加えて、**3 ヶ月使って発火回数 0 のガードは削除**してください。「念のため」だけのガードは、リポジトリの動かしづらさだけが残ります。物理ガードは「効いている証拠」を出し続けないと棚卸し対象です。

## まとめと次の記事

このスクリプトで防げるのは **「ソフト言語を含む新規 commit」だけ** です。既存のドキュメントに既に紛れ込んでいる「気をつける」「注意する」は別タスクで grep して洗い出す必要があります（`grep -rnE '気をつけ|注意する|できるだけ' docs/` を一度回せば、棚卸しリストが手に入ります）。

「気をつける」を書けなくしただけで、ルール文書の質は明らかに上がります。書こうとした瞬間に reject されるので、書く側が **「これは具体的に何をやる行動か？」を考えざるを得なくなる** のが、このガードの一番の効果です。テキストで守らせるのではなく、書く側の思考プロセスを変えてしまう ─ それが物理ガードの面白いところです。

次の記事は、**並走セッション問題** を扱います。複数の Claude セッションが同じファイルを同時に編集して、片方の変更がもう片方に上書きされる事故を、`WORKING.md` + `check-concurrent-sessions.sh` で物理ブロックする運用設計です。
