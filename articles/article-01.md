## 同じ事故を 2 回踏んだあなたへ

Claude Code を 2〜3 週間使ってきて、最近こんな経験をしていませんか。

- 「本番 URL を確認してから完了報告すること」と `CLAUDE.md` に書いたのに、CI が green になっただけで「完了しました」と報告される
- 「同じファイルを 2 つのセッションで触らない」と書いたのに、別タブの worktree が普通に上書きしている
- 「気をつけてください」と先週書いた事故が、今週まったく同じ形で再発する

1 回目は「自分の指示が悪かった」と思えるんですよね。2 回目になると「あれ、ルール書いても効かないな」とザワッとする。**この記事は、ちょうどその違和感が湧いてきたタイミングのあなた向け**に書いています。

結論から書きます。その感覚は正しいです。`CLAUDE.md` にいくら丁寧にルールを書いても、Claude はそれを守りません。これはあなたの書き方が悪いのでも Claude が悪いのでもなく、**「テキストで縛る」設計が壊れている**のが原因です。解決策は「ルールをテキストで書くのをやめて、物理的に違反不能にする」こと。本記事ではこれを「物理ガード」と呼びます。

---

## なぜ「気をつける」が効かないのか

LLM は instructions を「文字通り狭く」解釈する性質があります。

人間にとっての「気をつける」は「文脈に応じて適切に判断する」ですが、LLM にとっての「気をつける」は「該当箇所に『気をつけた』フラグを立てて完了」になりがちです。判断の精度を上げる呪文ではなく、ただのチェックボックスとして処理されます。

実話を 1 つ汎用化して紹介します。

> あるサービスでスケジューラーが失敗し続け、ユーザーから見えるデータが 12 時間古いまま放置されたことがありました。原因はコード修正が「push されずに滞留」していたこと。あるセッションがコードを編集したのに `git push` し忘れたまま終わり、本番に届かなかったのです。
>
> このとき `CLAUDE.md` には「セッション終了前に push を確認すること」と書いてありました。ですが Claude は「コードを変更したセッション」を毎回正確に分類できません。書いたルールが「人間の判断」を要求している時点で、再発防止になっていなかったわけです。

ルールを書いたとき、人は「これで再発を防げる」と感じます。ですが「ルールを書いた」と「再発を防いだ」は別の事象です。**機械的に強制する仕組みを伴わないルールは、ただのチェックボックス**でしかありません。

---

## 解決策: 物理ガード

「気をつける」を、**「書けなくする・触れなくする・実行できなくする」** に振り切ります。Git の hook や CI、SLI を使えば、Claude が `CLAUDE.md` を読み忘れても、別セッションが暴走しても、機械的に効きます。

| ソフト規律（壊れる） | 物理ガード（壊れない） |
|---|---|
| 「気をつける」と書く | `pre-commit` hook で「気をつける」を検出 → コミット reject |
| 「同じファイルを 2 セッションで触らない」と書く | `WORKING.md` 宣言なしの編集を `pre-commit` で reject |
| 「本番確認してから完了報告」と書く | `done.sh` が curl で本番 URL を叩き、200 以外なら exit 1 |
| 「`main` に直 push しない」と書く | `pre-push` hook で `refs/heads/main` への push を物理 reject |

発想は「Claude を信頼する」のではなく「**Claude を信頼できる状態に物理的に保つ**」です。

### 実例 1: 「気をつける」を pre-commit で reject する

`.md` の diff に対してパターンマッチをかけ、`気をつける` / `注意する` / `できるだけ` / `なるべく` を検出したらコミットを止めます。

```bash
#!/bin/bash
PATTERNS='気をつける|注意する|できるだけ|なるべく'
for f in $(git diff --cached --name-only --diff-filter=ACM | grep '\.md$'); do
  if git diff --cached "$f" | grep -E "^\+.*($PATTERNS)" > /dev/null; then
    echo "❌ '$f' にソフト言語が含まれています。物理ガードに置き換えてください。"
    exit 1
  fi
done
```

これを `.git/hooks/pre-commit` に仕込むと、「気をつける」と書こうとした瞬間に物理的に止まります。**書けないものは規範になりません。**

### 実例 2: `WORKING.md` で並走を物理ブロックする

別 worktree で起動した別セッションの存在は Claude からは見えません。なので「いま誰がどのファイルを触っているか」を `WORKING.md` に 1 行ずつ宣言させて、`pre-commit` が宣言なしの編集を検出 → reject。さらにセッション起動時に `git worktree list` を見て、アクティブな worktree が 2 件以上なら起動を物理ブロックします（脱出口として `ALLOW_CONCURRENT_CODE=1` を残すのは大事）。

### 実例 3: `done.sh` で本番 URL を確認させる

完了報告用のスクリプトの中で本番 URL を curl で叩き、HTTP 200 が返った時刻を `Verified: <URL>:200:<timestamp>` の形式でコミットメッセージに必須追加させます。`commit-msg` hook で `feat:` / `fix:` 系のコミットに `Verified:` 行が無ければ reject。「**本番確認しないと commit が物理的に書けない**」状態を作ります。

## 効果と限界（先に言っておきます）

正直な数字を出します。自分のプロジェクトで 30 件の失敗を観測した結果、物理ガードで防げたのは**約 83%、完全に再発ゼロになったのは 40% 程度**でした。残りはガード設計自体の穴・人間判断が必要な領域です。万能ではありません。

それから、書いておくべき正直な話がもう 1 つあります。**Anthropic 公式が hooks API や sub-agents を急速に拡充しているので、本記事のアプローチは公式機能に内包されていく可能性が高い**です。半年〜2 年で「公式 + テンプレ集で十分」という世界になるかもしれません。

それでも今このアプローチを書く価値があると判断したのは、**事故が起きるのは今だから**です。公式に内包されるのを待っている間に、データは消え、ユーザーは離れ、信頼は壊れます。物理ガードは「公式が組み込むまでの繋ぎ」と「自分のプロジェクトに合わせてカスタマイズする土台」として使えます。

## 今日から試せる：3 ファイルから始める

## さらに深く対策したい人へ：BOOTH フルキット

3 ファイルで「効いた」と感じたら、**フルキット**に進むと範囲が一気に広がります。本記事中で省略した内容を全部入れてあります。

- **30 件の実失敗記録**（`lessons-learned.md`）— ある SaaS プロジェクトで実際に踏んだ事故と、それを物理化した対策のペア。事故の現象 → 根本原因 → 物理ガード実装 → 効果計測まで 1 件ずつ書いてあります
- **CI workflow 一式**（GitHub Actions）— `check-soft-language` / `check-concurrent-sessions` / `freshness-check` / `pii-scan` など、コミット時だけでなく PR レビュー時にも物理ガードが効く構成
- **git hook 一式**（pre-commit / pre-push / commit-msg）— 本記事の実例 1〜3 の完全版
- **詳細ルール集**（`bug-prevention.md` / `design-mistakes.md` / `global-baseline.md`）— 30 件の失敗から抽出した、汎用化済みのアンチパターン
- **セットアップ手順書**（30 分で完了）

価格は **¥2,980**（買い切り・MIT ライセンス）。「最強テンプレ」ではなく「**30 件失敗して作った、不完全だが効くキット**」です。BOOTH ページは準備中で、公開後 OSS リポの README からリンクします。
