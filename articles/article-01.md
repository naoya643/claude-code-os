## 結論から

Claude Code を 2〜3 週間使ってみて「言うことを聞かないな」と感じているなら、その感覚は正しいです。`CLAUDE.md` にいくら丁寧にルールを書いても、Claude はそれを守りません。これは指示の書き方が悪いのでも Claude が悪いのでもなく、**「テキストで縛る」設計が壊れている**のが原因です。

解決策は「ルールをテキストで書くのをやめて、物理的に違反不能にする」こと。本記事ではこのアプローチを「物理ガード」と呼び、3 つの実例とともに紹介します。

---

## 「言うことを聞かない」と感じる瞬間

Claude Code を触り始めて 1 週目は感動するはずです。`CLAUDE.md` にプロジェクトの約束事を書いておけば、Claude が文脈を踏まえて働いてくれる。最初は本当によく動きます。

ところが 2 週目から違和感が出てきます。

- 「本番 URL を確認してから完了報告」と書いたのに、CI が green になっただけで「完了しました」と報告される
- 「同じファイルを 2 つのセッションで同時に触らない」と書いたのに、別タブの別 worktree が普通に同じファイルを上書きしている
- 「気をつけてください」と書いた翌週に、同じ事故が再発する

ここで多くの人は「自分の指示が下手なのかも」と思って `CLAUDE.md` を書き足します。が、ルールを増やしても効きません。むしろ増やすほど守られなくなります。

「文字で書いたルールを LLM が守ってくれる」という前提自体が、この時点で崩れています。

---

## なぜ「気をつける」が効かないのか

LLM は instructions を「文字通り狭く」解釈する性質を持ちます。

人間にとっての「気をつける」は「文脈に応じて適切に判断する」ことですが、LLM にとっての「気をつける」は「該当箇所に『気をつけた』フラグを立てて完了」になりがちです。判断の精度を上げる呪文ではなく、ただのチェックボックスとして処理されるのです。

実話を 1 つ汎用化して紹介します。

> あるサービスでスケジューラーが失敗し続け、ユーザーから見えるデータが 12 時間古いまま放置されたことがありました。原因はコード修正が「push されずに滞留」していたこと。あるセッションがコードを編集したのに `git push` し忘れたまま終わり、本番に届かなかったのです。
>
> このとき `CLAUDE.md` には「セッション終了前に push を確認すること」と書いてありました。ですが Claude は「コードを変更したセッション」を毎回正確に分類できません。書いたルールは、人間か LLM の判断を要求している時点で再発防止になっていませんでした。

「気をつける」というルールは、判断を人間と LLM に押し付けているだけで、機械的に強制する仕組みを一切持っていません。それなのに「書いた = 対策済」と錯覚してしまうのが一番の罠です。

---

## 解決策: 物理ガード

ではどうするか。**「書けなくする・触れなくする・実行できなくする」** に振り切ります。Git の hook や CI、SLI を使ってルールを物理的に違反不能にしてしまえば、Claude が `CLAUDE.md` を読み忘れても、別セッションが暴走しても、機械的に効きます。

| ソフト規律（壊れる） | 物理ガード（壊れない） |
|---|---|
| 「気をつける」と書く | `pre-commit` hook で「気をつける」を検出 → コミット reject |
| 「同じファイルを 2 セッションで触らない」と書く | `WORKING.md` 宣言なしの編集を `pre-commit` で reject |
| 「本番確認してから完了報告」と書く | `done.sh` が curl で本番 URL を叩き、200 以外なら exit 1 |
| 「`main` に直 push しない」と書く | `pre-push` hook で `refs/heads/main` への push を物理 reject |

ポイントは、**「Claude を信頼する」のではなく「Claude を信頼できる状態に物理的に保つ」** という発想の転換です。

## 実例 3 つ

### 1. 「気をつける」を pre-commit で reject する

`.md` の diff に対してパターンマッチをかけ、`気をつける` / `注意する` / `できるだけ` / `なるべく` を検出したらコミットを止めます。

```bash
#!/bin/bash
PATTERNS='気をつける|注意する|できるだけ|なるべく'
for f in $(git diff --cached --name-only --diff-filter=ACM | grep '\.md$'); do
  if git diff --cached "$f" | grep -E "^\+.*($PATTERNS)" > /dev/null; then
    echo "❌ '$f' にソフト言語が含まれています。物理ガードに置き換えてください。"
    exit 1
  fi
done
```

これを `.git/hooks/pre-commit` に仕込むと、「気をつける」と書こうとした瞬間に物理的に止まります。書けないものは規範になりません。

### 2. `WORKING.md` で並走を物理ブロックする

別 worktree で起動した別セッションの存在は Claude からは見えません。なので「いま誰がどのファイルを触っているか」を `WORKING.md` に 1 行ずつ宣言させます。`pre-commit` が宣言なしの編集を検出 → reject。さらにセッション起動時に `git worktree list` を見て、アクティブな worktree が 2 件以上あれば起動を物理ブロックします（脱出口として `ALLOW_CONCURRENT_CODE=1` を残すのは大事）。

これで「並走する物理的可能性がゼロ」になります。

### 3. `done.sh` で本番 URL を確認させる

完了報告のスクリプトの中で本番 URL を curl で叩き、HTTP 200 が返った時刻を `Verified: <URL>:200:<timestamp>` の形式でコミットメッセージに必須追加させます。`commit-msg` hook で `feat:` / `fix:` 系のコミットに `Verified:` 行が無ければ reject。

「本番確認しないと commit が物理的に書けない」状態を作ります。「CI が green だから大丈夫だろう」が通用しなくなります。

---

## 限界も正直に書いておく

物理ガードは万能ではありません。経験上、事故の 8 割は防げますが、残り 2 割は「ガード自体の設計に穴がある」「人間の判断が必要」という領域です。`ALLOW_CONCURRENT_CODE=1` のような bypass を残さないと、本当に並走させたい場面で詰みます。

それから、Anthropic 公式が hooks API などを拡充しているので、本記事のアプローチは公式機能に内包されていく可能性が高いです。それは歓迎すべきことで、本リポジトリの価値は「公式が組み込むまでの繋ぎ」と「自分のプロジェクトに合わせてカスタマイズする土台」だと考えています。

---

## まとめ

- `CLAUDE.md` に「気をつける」と書いても効かない。LLM はルールを狭く解釈するので、書いた瞬間に「対策済」と錯覚してしまうだけ
- 解決策は **物理ガード**: hook / CI / SLI でルールを違反不能にする
- 最小構成は 3 ファイル（`CLAUDE.md` / `WORKING.md` / `done.sh`）から始められる

次回は「『気をつける』と書いても意味がない理由」をさらに掘り下げ、`check-soft-language` の実装と運用ノウハウを書く予定です。
