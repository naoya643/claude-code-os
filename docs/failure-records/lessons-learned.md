# lessons-learned.md — 失敗記録 (空テンプレート)

このファイルは **append-only** の失敗記録です。**消さない・並べ替えない**。
あなたのプロジェクトで実際に起きたインシデントを 1 件ずつ追記していきます。

> 同じバグを 3 ヶ月後にまた踏まないための、唯一の防御は「過去の失敗を構造化して残すこと」です。
> このファイルが厚くなるほど、Claude Code はあなたのプロジェクト固有の地雷を踏まなくなります。

---

## このファイルの育て方

インシデント（本番障害・データ不整合・PR レビュー指摘・実機検証失敗）が起きたら、
**24 時間以内**に以下のテンプレを 1 件追加してください。

時間が経つと「なんで起きたか」が記憶から消え、書けなくなります。

---

## エントリのテンプレート

新しい失敗を記録するときは、このセクションをコピーして本ファイルの先頭付近に貼り付け、
最新が上に来るようにしてください（時系列降順）。

```
## YYYY-MM-DD <短い見出し>

**何が起きたか** (1-2 文):
  例: 本番デプロイ後に X 機能が動かなくなった。ユーザーから N 件の問い合わせ。

**なぜなぜ 5 段階** (Why1 は症状、Why5 は構造):
  Why1: <症状そのもの>
  Why2: <Why1 を引き起こした直接原因>
  Why3: <Why2 を許容した手順 / プロセス>
  Why4: <Why3 が存在した習慣 / 前提>
  Why5: <Why4 を成り立たせていた構造的問題>

**仕組み的対策**:
  ID: <YYYY-MM-DD-項目名>
  ① <CI / hook / metric / SLI / scripts のどれかで物理化する対策を 1 行>
     実装場所: <ファイルパス>
     状態: ✅ 実装済 / 🚧 着手中 / ⏳ 未着手

  ② (必要なら 2 件目)
     ...

**思想ルール** (物理化困難な場合のみ):
  - <人間 / Claude が「気をつけて」では守れないが、構造を変えれば守れるルール>
  - 物理化候補があれば書き出しておく（後で実装する）

**関連 PR / commit**:
  - <PR URL or commit SHA>
```

---

## ⚠️ 「気をつける」だけの対策は書かない

「仕組み的対策」セクションに以下のようなソフト言語を書くと、`scripts/check_soft_language.sh` が
ビルドを落とします:

- 「気を付ける」「気をつける」
- 「注意する」「意識する」
- 「確認する」（手順チェックリストにも書かない・物理検証スクリプトに落とす）

**理由**: これらは「願い」であって対策ではありません。LLM はストレス下で忘れます。
人間もオンコール対応で忘れます。**仕組みが守るものだけが、本当に守られる**。

代わりに以下の形に落としてください:

| 形 | 例 |
|---|---|
| CI ゲート | `.github/workflows/<name>.yml` で違反検知 → ビルド fail |
| git hook | `.git/hooks/pre-commit` で commit 拒否 |
| metric 閾値 | CloudWatch / Datadog で SLI 違反アラート |
| scripts | `done.sh` / `session_bootstrap.sh` に物理検証ステップ |
