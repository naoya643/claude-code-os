> このファイルを参照するタイミング: タスク記述前（finder）・修正実装前（implementer）に必ず全行確認する。

## このファイルへの追加ルール

| 追加するタイミング | 設計の前提が間違っていたと判明したとき |
|---|---|
| **追加する内容** | 「機能名」「想定していたこと」「実際に起きたこと」の 3 列 |
| **追加しない** | 単発のバグ。バグは `bug-prevention.md` へ |
| **粒度** | 「外部システムの挙動を勘違いしていた」「ユーザー体験の前提を勘違いしていた」など、**思考の前提**レベルのミス |
| **削除** | このファイルは原則 append-only |

## 設計ミスパターン

| 機能 | 設計の前提（間違い） | 実際 |
|---|---|---|
| <!-- 例 (汎用) --> 外部 API キーワード検索 | 検索クエリ「テクノロジー」で取得したフィードは全部テクノロジー記事 | API が別ジャンルの記事を混入させる。フィードのジャンルラベルは信用できない |
| <!-- 例 (汎用) --> アフィリエイト配置 | ニュース見出しを商品検索クエリにすれば収益になる | 「肝臓がんリスク」で Amazon 検索になる。ニュース閲覧中ユーザーは商品を買いに来ていない。文脈マッチがないと逆効果 |
| <!-- 例 (汎用) --> Slack 通知 | secret 名が合っているはず | `SLACK_WEBHOOK_URL` vs `SLACK_WEBHOOK` で長期間 404。env var 名は CI で参照箇所と定義箇所を突合せよ |
| <!-- 例 (汎用) --> 外部投稿 API | 設定済みのはず | env var 未設定で一度も投稿できていなかった。CI で「主要環境変数の存在確認」を必須化 |

## なぜ design-mistakes と bug-prevention を分けるのか

| | bug-prevention.md | design-mistakes.md |
|---|---|---|
| 粒度 | 1 関数 / 1 行のロジック | 機能全体の設計前提 |
| 修正方法 | 該当箇所を直す | 設計を作り直す |
| 再発防止 | grep で同類検索 | 似た「前提」を持つ機能を全部見直す |
| 例 | 「null チェック忘れ」 | 「外部 API のフィードはジャンル精選されている」という前提 |

設計ミスは 1 ファイル単位の修正では再発しない。「**同じ前提を持つ別機能**」にも横展開して見直す必要がある。だから別ファイルに分けて、設計判断のチェック時に必ず通る場所にする。
