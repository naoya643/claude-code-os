# バグ再発防止ルール（再発させない）

> このファイルを参照するタイミング: タスク記述前（finder）・修正実装前（implementer）に必ず全行確認する。
>
> **このファイルは育てる前提のテンプレートです。** バグを直すたびに 1 行追加してください。

---

## このファイルへの追加ルール

| 追加するタイミング | バグを 1 件直したとき |
|---|---|
| **追加する内容** | 1 フレーズで言える「パターン名」と、1 文の「対処ルール」 |
| **追加しない** | 「気をつける」「注意する」だけの対策。CI / hook / script で物理化できないか先に検討 |
| **粒度** | 表 1 行に収まる範囲。長文の説明が必要なら別ファイルへ |
| **削除** | このファイルは原則 append-only。古いルールでも残す（再発防止の証跡） |

## バグパターン

| パターン | ルール |
|---|---|
| <!-- 例 (汎用) --> **環境変数名の typo** | secret 名・env var 名は CI で参照箇所と定義箇所を突合チェックする。「設定済みのはず」を信じない |
| <!-- 例 (汎用) --> **API URL のスラッシュ重複** | `BASE_URL` は末尾スラッシュの有無を統一する。CI で `\\+'\\/'` のような重複連結を検出 |
| <!-- 例 (汎用) --> **失敗時に「処理済み」フラグを立てる** | 成功時のみフラグ True。失敗時 True → 永遠に再処理されない事故が起きる |
| <!-- 例 (汎用) --> **エラー無視には理由の明文化必須** | エラー・警告をスキップ・無視する場合は「なぜ無視して良いか」をコメントか PR 説明に必ず書く。「おそらく大丈夫」は理由にならない |
| <!-- 例 (汎用) --> **1 バグ発見 → 同類全探索ルール** | バグを 1 件修正したら「同じパターンが他にないか」をコードベース全体で grep してから止まる。確認証跡を commit message に書く |
| <!-- 例 (汎用) --> **再発防止策は「他でも起きないか」横断確認必須** | 修正後のなぜなぜ分析で「仕組み的対策」を書くとき、同じ根本原因が別ファイルにも潜在しないか確認する |
| <!-- 例 (汎用) --> **タスク実装前「ユーザー体験で何が変わるか」必須確認** | 実装着手前に「このタスクはユーザーに何を変えるか」を 1 行で書けること。書けないタスクは着手しない |
| <!-- 例 (汎用) --> **コスト増加実装は即スキップ** | API コール回数増加・外部 API 追加呼び出し・新規有料リソース作成など、コストが上がる実装は実施前に必ず停止してオーナーに確認 |
| <!-- 例 (汎用) --> **無限ループ・暴走リスク禁止** | リトライループ・再処理ループを実装する場合は必ず上限回数と脱出条件を明示する。上限なしの while ループ・再帰は禁止 |

## このパターンが育つと何が起きるか

3 ヶ月運用すると、表が 20〜30 行になる。
半年運用すると、表が 50 行を超える。

その状態で:
- 「タスク記述前に全行確認」が物理的にできる（30 行は数分で読める）
- 新規バグの 60〜70% は「既知パターンの再発」になり、ここを引けば未然防止できる
- Claude が「修正実装前にこの表を見る」のがフローに乗ると、根本原因を踏まなくなる

**1 行追加の習慣だけ守れば、自動的に最大の資産になる**。
